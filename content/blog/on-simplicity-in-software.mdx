I've been writing software professionally for years now, and if there's one lesson that keeps asserting itself, it's this: **simplicity is not the starting point — it's the destination**.

## The Complexity Trap

Every developer has been there. You start building a feature, and within an hour you're designing an abstraction layer for a problem you don't have yet. You create a factory that produces builders that configure strategies. It's beautiful architecture. It's also completely unnecessary.

The trap is thinking that more abstraction equals better code. In reality, the best code is often embarrassingly straightforward.

## Three Rules I Follow Now

After years of over-engineering (and the painful refactors that followed), I've settled on three rules:

**1. Don't abstract until the third occurrence.** The first time you write something, write it. The second time, notice the duplication but resist the urge. The third time, *now* you have enough information to abstract correctly.

**2. Measure the cost of being wrong.** Before adding complexity, ask: "If this turns out to be the wrong abstraction, how hard is it to undo?" If the answer is "very hard," keep it simple until you have more information.

**3. Optimize for deletion.** Write code that's easy to delete. Small, independent modules with clear boundaries are easier to remove than deeply entangled abstractions.

## The Paradox of Simple Code

Here's what's counterintuitive: simple code often takes longer to write than complex code. It requires you to truly understand the problem, strip away the inessential, and find the core of what needs to exist.

As Blaise Pascal allegedly wrote: *"I would have written a shorter letter, but I did not have the time."*

The same applies to code. Simplicity is achieved through editing, not through first drafts.

## In Practice

This blog itself is an exercise in simplicity. It's MDX files in a Git repository, rendered by Next.js, deployed on Vercel. No database, no CMS admin panel, no authentication layer.

Could it be more featureful? Of course. But the question isn't what it *could* do — it's what it *needs* to do. And for a personal blog, the answer is: render words on a screen, beautifully and fast.

That's it. And that's enough.
